# team-knowledge ナレッジ管理ガイド

このリポジトリはチーム共有のナレッジハブです。
AIはこのガイドに従ってナレッジの追加・更新・整理を行ってください。

## リポジトリ構成

```
company/          # 全社共通知識（規定・ルール・用語・技術スタック等）
skills/
  approved/       # 承認済みスキル
  proposals/      # 提案中のスキル（PRで承認後にapproved/へ移動）
best-practices/   # Claude活用のベストプラクティス・Tips
teams/
  registry.md     # チーム専用リポジトリへのリンク集
source/           # [Actions用] ナレッジ投入口（処理後に削除される）
```

## ナレッジ追加フロー

### Phase 1: 受付・分類
- 追加したい情報の内容を理解する
- 対象ディレクトリを判断する：
  - 会社の決まり事・規定・共通情報 → `company/`
  - Claude活用のTips・ワークフロー → `best-practices/`
  - スキルの共有 → `skills/proposals/`
  - チーム固有の知見 → `teams/registry.md` を参照し、該当チームリポジトリへ誘導

### Phase 2: 探索
- 対象ディレクトリのファイル一覧と各ファイルの行数を取得する
- サブエージェントに各ファイルをReadさせる
  - 読めた場合：今回の追加内容に関連する部分だけ要約して返してもらう
  - エラー（トークン超過）の場合：そのファイルを「分割が必要」としてフラグする
- 関連ファイルの全体像を把握する

### Phase 3: 計画提示
- 変更計画を提示する：
  - どのファイルをどう変更するか
  - 新規ファイルを作るか
  - 分割が必要なファイルがあればその計画も含める
- **スキル版**: ユーザーに確認を求める
- **Actions版**: 計画をPRの説明に記載して自動実行する

### Phase 4: 実行
- ファイルを編集・作成する
- 編集後のファイルをサブエージェントにReadさせてサイズ確認する
  - 読めた → OK
  - エラー → トピック単位で分割を実施する
- ブランチを作成し、変更をコミットしてPRを作成する

### Phase 5: 完了
- PR URLを共有する
- source/ に投入ファイルがあった場合は処理済みとして削除する

## コンテンツ種別ごとのAIの振る舞い

### company/（全社共通知識）
- AIが積極的に情報を整理・統合する
- 既存ファイルとの重複や矛盾を確認し、適切に追記・更新する
- 情報のリファクタリング（再構成・分割・統合）も行う

### skills/, best-practices/
- ユーザーが提供した内容をそのまま登録する
- AIは中身の加工をしない
- 以下のみ提案する：
  - ファイルサイズが大きすぎる場合の分割
  - フォーマットの提案（強制ではない）

## ファイル管理ルール

### ファイルサイズ
- 固定の行数制限は設けない
- サブエージェントにReadさせて読めるかどうかで判断する
  - 目安：Sonnet/Opusで約10,000トークン（日本語約800行、英語約1200行）
- 超過した場合はトピック単位で分割する

### ファイル命名
- 日本語ファイル名は避け、英語のkebab-caseを使う
  - 例: `deployment-rules.md`, `react-patterns.md`
- 内容が明確にわかる名前をつける

### フォーマット
- Markdownを使用する
- 見出し（##）でセクションを区切る
- 各ファイルの先頭に、そのファイルの概要を1行で書く
